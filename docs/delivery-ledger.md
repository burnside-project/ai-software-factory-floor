## Known backlog: pre-existing `validate-artifacts.sh` failures

**Recorded by TICKET-066 (SPEC-016, AC2b). Not fixed there — recorded so the backlog is
not silently inherited.**

Running `bash templates/factory/scripts/validate-artifacts.sh` from the repo root reports
**42 errors, all on committed files**:

| Count | Location | Cause |
|---|---|---|
| 41 | `tickets/completed/` | Older prose-style tickets (`# Ticket: …` / `Ticket ID: TICKET-NNN`) with no YAML front-matter. `validate-artifacts.sh:26-28` requires a front-matter `id:`. |
| 1 | `specs/completed/SPEC-013b-provision-orchestrator` | The deliberate `013b` suffix does not match the validator's `SPEC-NNN-<slug>` directory regex. |

**Why this went unnoticed.** `validate-artifacts.sh` is not wired into this repo's own
`ci.yml` or `Makefile` — the factory does not gate *itself* on it. The script is a
*template* installed into delivery repos, where `setup-repo.sh` makes `factory-naming` a
required check. So the factory can carry a large validator backlog without ever seeing red.
That same blind spot is how the authoring templates drifted out of the convention for as
long as they did (TICKET-066).

**Decision: ACCEPT for now, do not fix.**

- The 41 ticket files are **completed** artifacts. Rewriting historical records to satisfy a
  gate they predate changes the archive to please a linter — the wrong trade. Newly authored
  tickets pass, and TICKET-066's template fix plus `tests/factory/test-templates-validate.sh`
  keep it that way going forward.
- `SPEC-013b` is a real, intentional identifier (a follow-on increment to SPEC-013). The
  *validator regex* is the thing that is wrong here, not the directory name. Widening it to
  allow a letter suffix is a one-line change, but it alters a script that gates every
  provisioned repo, so it belongs in its own thin spec with its own review — not smuggled in
  under a template-repair ticket.

**Revisit if** either becomes true:
1. `validate-artifacts.sh` is ever wired into this repo's CI — the backlog must be cleared
   or explicitly scoped first, or the check lands permanently red.
2. A *new* artifact appears in this list. That means the guard has failed and the drift has
   restarted; fix it immediately rather than growing the backlog.

A permanently-red validator trains operators to ignore it. That is the actual risk being
accepted here, and it is why item 2 above is a tripwire rather than a preference.
