# ADR-0015: Factory configuration authority — authored YAML is the source, shell is the runtime

Status: **Proposed** (aligned to SPEC-019 revision 2 — the three forks are now decided; pending
SPEC-019 acceptance)
Date: 2026-07-20
Deciders: Architect (via SPEC-019)
Related: SPEC-019 (factory config productization), ADR-0006 (data-driven ruleset selection —
this ADR is a deliberate, justified departure/extension), ADR-0012 (`provision.sh`
orchestrator), SPEC-013b, SPEC-017/018 (downstream consumers of resolved config)

## Context

The repository fuses a generic *framework* with *one instance's configuration*
(dataalgebra-engineering's org, the "Burnside Software Factory" board, `burnside-project-*`
naming, self-hosted runners, repo-specific rulesets). The portability audit counted 56
instance-specific literals across framework code. A second repo or a new org cannot adopt the
framework without editing framework code, and the failure is silent: forget to override
`${ORG:-dataalgebra-engineering}` and board-sync writes to *our* board and reports success
(`board-sync.sh:12`, `metrics.yml:31-32`, `setup-project.sh:15`, `audit-org-naming.sh:9`).

SPEC-019 introduces one authored configuration file as the single source of truth. This ADR
records the *durable* contracts that follow — the parts that outlive any one ticket. An earlier
draft left three forks (lock file, declarative subject, precedence/scope) open; SPEC-019
revision 2 decided them, and Decisions 6–8 below record those resolutions.

## Decision

### 1. The checked-in instance configuration is authoritative for all non-secret config

`factory.config.yaml` (authored, versioned, hand-edited) is the single source of truth for all
non-secret configuration. GitHub variables, project identifiers, ruleset assignments, and
generated runtime files are **outputs, not competing sources of truth.** No script default may
point "home": there is no `${ORG:-dataalgebra-engineering}` fallback after cutover — a missing
required value fails closed.

### 2. Secrets are references, never values

The config declares *which* secret must exist (by name); the secret material lives only in
GitHub. `factory` verifies presence and never sets or reads secret values. (Secret *scope* —
org vs repo — must be named in the schema; see Open Fork C.)

### 3. Authoring format ≠ runtime format — a justified departure from ADR-0006

ADR-0006 chose TSV over JSON to keep the hot path jq-free and Bash-3.2-native, and to keep
"add a repo = add a line" honest. This ADR **preserves that rationale** while adding a schema
and versioning that a flat TSV cannot carry:

- The `factory` CLI (validate/sync/migrate/doctor/inspect) MAY require python3 + PyYAML.
- **Workflows and hot-path scripts MUST NOT parse YAML.** They read only compiled, shell-native
  outputs (`factory.env`, TSV) and GitHub variables (`vars.*`), staying Bash-3.2 and
  parser-free.
- The `apiVersion` group is **framework-neutral** (`factory.sh/v1`, not
  `factory.dataalgebra.dev/...`): the schema that de-org-ifies the framework must not carry the
  org in its own envelope. Instance identity lives in `metadata`/`spec`.

**The compile step is a first-class, contracted artifact-generator, not an implicit side
effect.** `factory` compiles `factory.config.yaml` → the runtime outputs. Those outputs are
generated artifacts, regenerated by an explicit step, and guarded by a drift check that asserts
the compiled output matches the authored YAML (the same posture as a committed lockfile checked
for drift in CI). This closes the two-source drift the whole effort exists to remove: if a
compiled `factory.env` could be edited independently of `factory.config.yaml`, the drift would
return. (SPEC-019 must ticket this compile step explicitly, inside the MVP, between the loader
and the fail-closed cutover.)

**GitHub variables are a legitimate runtime output of the author→push direction.** Runner labels
already reach workflows via `vars.RUNNER_LABELS` (`metrics.yml:25`), read directly with
`fromJSON`. This is workflow-native and is retained. The "no YAML in the hot path" rule is
therefore stated as **"no workflow or hot-path script parses YAML,"** not "workflows read only
`factory.env`/TSV" — the latter is already false and would force a needless rewrite of a working
pattern.

### 4. Three directions of truth

- **author → push** (org slug, runner labels, naming policy, ruleset routing): pure derivation
  from `factory.config.yaml` into GitHub variables, rulesets, and compiled outputs.
- **external → verify-only** (App private key, DeepSeek key): the config references them; sync
  checks presence, never handles material.
- **GitHub → resolve, do not cache** (project number/node ID, ruleset IDs, org ID): minted by
  GitHub, resolved by *natural key* on each run, cached nowhere (Decision 6). The single value
  with no cheap natural key is the App **installation ID**; if ever cached it is the only thing a
  minimal state file holds, and it carries no credential material.

### 5. Consumer zero and config-root discovery

The framework repo expresses its own instance through the same schema it ships and deletes its
literals against it: if the factory cannot express itself, no consumer can.

**Engine discovery and config discovery are distinct anchors.** The proven two-layout probe
(`metrics.yml:41-43`, `provision.sh:47`, `setup-project.sh:20`) resolves the *engine* from
`BASH_SOURCE`, never `$PWD`. The *consumer's* `factory.config.yaml`/`.factory/` lives at the
**consumer repo root**, which is not `BASH_SOURCE`-relative. Config discovery therefore uses a
separate consumer-root anchor (`git rev-parse --show-toplevel` / explicit `--config`), and:

- **The consumer's config always wins.** The vendored bundle MUST ship **no** instance config —
  only the schema and a clearly-marked example that lives outside the vendored tree — so a
  vendored consumer can never silently read the framework author's config.
- Consumer-zero (where both anchors coincide) **cannot prove correct config-root discovery.**
  Only the second-repo acceptance proof (SPEC-019 AC1) exercises the divergent-anchor path.

### 6. No persistent lock file — stateless natural-key resolution *(resolves former Fork A)*

The entire live runtime resolves GitHub objects by natural key — project by **title**
(`setup-project.sh:28`, `board-sync.sh:37`, `metrics.sh:40`), ruleset by **name**
(`setup-repo.sh:252`, `ruleset.pg-cdc.json:3`), org by **slug** — and caches no IDs. A persistent
lock re-introduces state the runtime deliberately avoids, and its only distinct consumer would be
a `sync --check` exit code 3 ("stale cache"), a distinction natural-key resolution makes moot (a
recreated board is simply re-resolved by title). **Decided:** ship **no persistent lock file** by
default; `--check` returns 0/1/2 only. The single value with no cheap natural key is the App installation ID; if it
is ever cached, that file holds only that ID (no authored field, no credential material — AC8),
is excluded from the vendored consumer bundle, and is updated incrementally per resolved resource
so a partial `--apply` cannot leave it inconsistent. (SPEC-019 revision 2 ticket 9; drops rev-1
ticket for a full `factory.lock.yaml` and exit code 3.)

### 7. Declarative subject = org policy + routing-named repos, not the whole fleet *(resolves former Fork B)*

The factory configures many repos in one org, but `provision.sh` takes the repo as an *argument*
(`provision.sh:87`) and `ruleset-map.tsv` names only the tailored repos. A declarative `sync`
needs a defined subject; it cannot reconcile "actual vs desired" for repos config does not
enumerate. **Decided:** `factory.config.yaml` describes **org-level policy + ruleset routing**
for v1; the managed repo set is **not** desired state this increment; per-repo bring-up stays
`factory provision <repo>` (imperative). `sync` reconciles org-scoped objects (project, org
variables) and the routing-named repos only — which also bounds the provision/sync convergence
(SPEC-019 ticket 11): the shared declarative surface is org + named repos, not the whole fleet.
Fleet-level desired state is a later spec.

### 8. Config is authoritative; secret scope is named in the schema *(resolves former Fork C)*

- **Precedence:** the file is authoritative; there is **no per-field env override** (`ORG=…`
  cannot override config). The only sanctioned escape for tests/CI is a whole-config pointer
  (`FACTORY_CONFIG=<fixture>`), never a per-field env, and never for identity.
- **Secret scope:** the schema states org-secret vs repo-secret for each referenced secret so
  `doctor`/`sync` know where to check presence (blast-radius is a security concern). Values are
  never handled (Decision 2); `validate` additionally **refuses secret-shaped values** so a
  secret cannot be pasted into the authored config by mistake.

## Consequences

Positive:
- One authored source; GitHub-side state is derived and reproducible.
- ADR-0006's Bash-3.2/parser-free hot path is preserved and extended with schema + versioning.
- Silent "writes to the wrong org and reports success" becomes a fail-closed error.
- The framework becomes adoptable by a second repo with zero framework-code edits (the headline
  proof), *provided* config-root discovery (Decision 5) is honoured.

Negative / trade-offs:
- A compile step is a new moving part; it must be guarded by a drift check or it recreates the
  two-source problem.
- Config-root discovery adds a second anchor discipline distinct from the engine probe; getting
  it wrong fails only on a real second repo, not in consumer-zero.
- The declarative subject is deliberately partial (org policy + routing-named repos, not the whole
  fleet); fleet-level desired state is left to a later spec. `sync` cannot reconcile repos config
  does not enumerate — imperative `provision <repo>` remains the bring-up path.

## Alternatives considered

- **Keep scattered `${ORG:-…}` env defaults.** Rejected: the silent-wrong-org failure mode and
  the 56-literal drift surface are the problem being solved.
- **One monolithic format (YAML end-to-end, workflows parse YAML).** Rejected: violates
  ADR-0006's Bash-3.2/parser-free hot path and adds a YAML parser to every workflow.
- **Resolved IDs in GitHub variables instead of a lock.** Rejected: variables *are* GitHub-side
  actual state, so caching GitHub IDs in variables removes any offline reference and cannot
  support the staleness distinction — strictly worse than either a committed lock or no lock.
- **Cache all GitHub IDs in a committed lock by default (as first drafted in SPEC-019).**
  Rejected (Decision 6): the runtime already resolves every object by natural key and caches none,
  so a lock re-introduces state the runtime avoids and earns only a `--check` exit-3 distinction
  that natural-key resolution makes moot. Default: no lock.
