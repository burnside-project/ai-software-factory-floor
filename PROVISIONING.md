# Provisioning — the single canonical onboarding runbook

Take a repo from zero to a **gated, delivering** factory instance. This is the one door:
it owns the end-to-end *ordering* (what to run, in what order, one-time-per-org vs.
per-repo) and **links** to the authoritative docs for each deep procedure rather than
restating them. Run everything below from a checkout of the **`ai-software-factory`**
repo unless a step says otherwise.

- **Path A — greenfield:** a brand-new repo the factory creates/owns end-to-end.
- **Path B — existing repo:** a repo that already has code/CI; the factory adds a
  reconciled gate layer without clobbering its pipeline.

The per-org steps are identical for both paths. The per-repo steps differ only where
noted (greenfield gets the shipped `ci.yml`/`deploy.yml`; an existing repo keeps its CI
and gets a *reconciled* ruleset).

> **Guided path:** `/provisioning:onboard-project` walks this runbook interactively (pausing for
> confirmation before each GitHub mutation) and `/provisioning:check-readiness` wraps its Step 0
> pre-flight. `/provisioning:onboard-repo` guides **Path B (existing repo)** — readiness →
> non-clobbering install → reconciled ruleset → hand-off to `/provisioning:activate-gates`.
> `/provisioning:activate-gates` guides the "Per repo" activation leg (steps 3–7)
> interactively for an already-bootstrapped repo. This runbook stays the single source
> of truth for the step detail.

> **Emerging configuration model (SPEC-019).** The instance-specific values this runbook
> currently sets via env vars and script edits (org, project title, runner labels, repo-naming
> policy, ruleset routing) are being consolidated into **one authored file, `factory.config.yaml`**
> — the single source of truth for all non-secret configuration, with GitHub-side state derived
> from it. The schema is [`templates/factory/schema/factory.config.v1.md`](templates/factory/schema/factory.config.v1.md)
> (`apiVersion: factory.sh/v1`); a starting point is [`docs/examples/factory.config.example.yaml`](docs/examples/factory.config.example.yaml).
> Authored config lives at the repo root; compiled/derived files live under `.factory/`
> (`.factory/ruleset-map.tsv` routing, `.factory/factory.env` generated by `factory sync`). The
> `factory` CLI that loads, validates, compiles, and reconciles this file lands across SPEC-019
> (TICKET-095…109); until then the env-var/script model below remains the operative path. Secrets
> are **references (`{ name, scope }`), never values** — the material stays in GitHub.
> The framework's **own** live instance is [`factory.config.yaml`](factory.config.yaml) at the repo
> root (consumer zero — it round-trips through `factory validate`); it is never vendored into a
> consumer bundle. A copy-to-adapt starting point is [`docs/examples/factory.config.example.yaml`](docs/examples/factory.config.example.yaml).

## Step 0 — Prerequisites / readiness

Run the read-only pre-flight from the factory repo root:

```bash
./scripts/check-readiness.sh
```

It checks that `gh` and `jq` are on PATH, `gh` is authenticated with the `project` scope,
and the `.claude/agents` + `.claude/commands` layers are present. Also install `openssl`
(used when minting the GitHub App JWT). Fix any `MISSING:` line before continuing.

Once the builder App exists (below), run the **cutover pre-flight** `factory doctor` to confirm the
credentials/App/dispatch chain end-to-end — it catches a missing App private key or an installation
that does not cover the target repos *before* a cutover, not during it. **`doctor` has a side
effect:** it issues one real `repository_dispatch` on the dedicated, unconsumed `factory-doctor-probe`
event (never a consumed type, so it moves nothing), and it never writes the App key or the minted
token to any output. It is an operator step, not run on-merge. See
[`docs/factory-cli.md`](docs/factory-cli.md#factory-doctor--cutover-pre-flight-ticket-107).

**Least-privilege credentials (do NOT mint a broad / all-scopes / `admin:org` / full-`repo`
classic PAT):**

| Credential | Scope | Set via |
|---|---|---|
| Operator `gh` CLI | `project,read:org,repo` | `gh auth refresh -s project,read:org,repo` (see [`ACTIVATE.md`](templates/factory/ACTIVATE.md)) |
| `PROJECTS_TOKEN` (metrics/board-sync PAT) | classic `project` — **read *and* write on org Projects v2**, and nothing else | repo secret; see [`METRICS.md`](templates/factory/METRICS.md) |
| Builder identity | least-privilege GitHub App (**not** an admin token) | [`RUNBOOK-github-app.md`](templates/factory/RUNBOOK-github-app.md) |

**Read the `PROJECTS_TOKEN` row carefully — it is not what an earlier version of this table
said.** That version described it as `project` plus a read-only repo scope, and asserted "no
write". Both halves were wrong. The classic `project` scope is read **and** write
(`read:project` is its read-only child), and `board-sync.yml:24` has always *written* the Stage
field with this token; the second scope it named does not exist in GitHub's classic-PAT scope list
at all. **Nothing has been elevated** — the token's scope is
unchanged and the description was corrected to match what it already was. Mint it with `project`
alone: it can move cards on the org board and it deliberately cannot touch repo contents.

**Credential split — why label creation does not use `PROJECTS_TOKEN`.** Creating labels needs
**repo write**, which `PROJECTS_TOKEN` does not have and must not be given. Label provisioning
therefore runs under **the operator's own `gh` auth** (already scoped `repo`, the row above) at
provisioning time, not from a workflow. Widening a repo secret that *any* workflow in the repo can
read — including one with an injection bug — to gain repo write would be a much larger exposure
than a one-time operator-run script.

**Set the `GITHUB_TOKEN` floor to read-only.** A workflow that omits a `permissions:` block does
**not** get a safe default — it inherits the repo's `default_workflow_permissions`, which in turn
inherits the org/enterprise default. On any org that has never been explicitly hardened (including
every org created before the Feb-2023 change) that inherited value is **read/write on every
scope**. With no key set at any level the effective scope of your workflows is externally
controlled. **No script sets this** — it is an operator step, once per org and once per repo:

```bash
# org floor (preferred — covers every repo, including ones provisioned later)
gh api -X PUT "/orgs/$ORG/actions/permissions/workflow" -f default_workflow_permissions=read
# per-repo floor (also set it here; a repo-level value overrides the org default)
gh api -X PUT "/repos/$ORG/<repo>/actions/permissions/workflow" -f default_workflow_permissions=read
```

This is defence in depth, not a substitute for per-workflow `permissions:` blocks: every shipped
workflow declares its own. The floor is what catches the one that forgets.

Secrets are **never** committed. Use env-var references / placeholders only, e.g.:

```bash
export ORG=<your-org>                 # e.g. an org you administer
export HOST_REPO=<host-repo>          # tracking/backup repo, e.g. roadmap
export SPEC_FILTER=<SPEC-ID-or-empty> # optional metrics filter, e.g. SPEC-005
export PROJECTS_TOKEN=<paste-at-runtime-not-in-a-file>
```

## Authoring `factory.config.yaml`

The factory is config-driven (SPEC-019): a single authored `factory.config.yaml` at your repo root
is the source of truth for all non-secret configuration; GitHub state is the derived output. You do
**not** edit framework code to adopt it.

1. **Copy the neutral example** and fill in your values — the schema documents every field:
   [`templates/factory/schema/factory.config.v1.md`](templates/factory/schema/factory.config.v1.md).
   Set `spec.github.organization`, `spec.github.project.title`, `spec.repositories.naming.allowedPatterns`,
   `spec.runners.labels`, and the `spec.identities.floor` refs (`appIdVariable`, `privateKeySecret`)
   as `{ name, scope }` pairs. Secrets are **named, never valued** here.
2. **Validate offline** — `factory validate` checks the schema and refuses a secret-shaped value
   pasted anywhere in the file.
3. **Pre-flight** — `factory doctor` (see Step 0) confirms the credential / App / dispatch chain.
4. **Plan, then apply** — `factory sync` prints a dry-run reconcile **plan** (zero mutation);
   `factory sync --apply` reconciles config → GitHub. See the credential matrix below.

Rotating any named secret later: [`docs/runbooks/secret-rotation.md`](docs/runbooks/secret-rotation.md).

### The sync credential matrix

`factory sync --apply` mutates live org state, so **who** may run it — and under **which** credential
— is load-bearing. Read-scoped verbs are CI-safe; the mutating apply is operator-only and is
**refused inside a workflow/CI context**, so a leaked workflow credential can never rewrite org state.

| Verb / mode | Scope | Credential | May run in a workflow / CI? |
|---|---|---|---|
| `sync` (plan), `sync --check`, `sync --check-compile`, `migrate`, `validate`, `inspect` | read-only | any read-scoped `gh` auth | **yes** (CI-safe) |
| `doctor` | **side effect** — one probe `repository_dispatch` | read-scoped + short-lived App token | **operator step** (not on-merge) |
| **`sync --apply`** | **mutates org variables / rulesets / project** | **operator-only, human-held admin `gh` auth** | **NO — refused** |

`sync --apply` is **never a workflow secret** and **never run on-merge**: a workflow/CI context
(`GITHUB_ACTIONS` / `CI`) is refused with a non-zero exit *before any mutation*. The builder App
deliberately lacks `Administration`, so `--apply` cannot run under it — an operator applies it under
their own admin auth. Full detail: [`docs/factory-cli.md`](docs/factory-cli.md).

## One-time per org

Run these **once per GitHub org**, as an org admin. They set up the shared identity and
tracking surface every repo in that org reuses.

1. **Builder App** — create + install the least-privilege GitHub App (Repository
   permissions only, **Workflows: write**, **no `Administration`**), then wire the git
   credential helper + bot commit identity. Full steps:
   [`RUNBOOK-github-app.md`](templates/factory/RUNBOOK-github-app.md).
   *Artifact:* a bot identity that authors PRs a human then approves.
2. **Teams / CODEOWNERS** — `templates/factory/scripts/setup-teams.sh` (add `--create` to
   create any missing team). **This must pass before you flip any ruleset to `active`**
   (C8): an unresolved / empty CODEOWNERS team makes GitHub treat that path as having **no**
   required reviewer — a silent **fail-open** of the review gate.
   *Artifact:* the org teams referenced by `CODEOWNERS` exist and are non-empty.
3. **Host / tracking repo** — `templates/factory/scripts/setup-host-repo.sh` (review the
   dry-run plan first, then `--apply`).
   *Artifact:* `$ORG/$HOST_REPO`, the backup + org-wide tracking home.
4. **Org Project** — `templates/factory/scripts/setup-project.sh`. Idempotent: a second run
   reports every field already present and mutates nothing.
   *Artifact:* the org Project (`Your Factory Board`; override with `PROJECT=`) with four
   script-provisioned fields:

   | Field | Type | Options |
   |---|---|---|
   | `Stage` | single-select | the ten Stage options from [`templates/factory/stage-map.tsv`](templates/factory/stage-map.tsv) |
   | `Spec` | text | free text, e.g. `SPEC-016` |
   | `Work Type` | single-select | `Spec`, `Ticket`, `Epic`, `Bug`, `Feature Request` |
   | `Priority` | single-select | `P0`, `P1`, `P2`, `P3` |

   > **The field is `Work Type`, never `Type`.** `Type` is a **reserved** Projects v2 field name;
   > the API rejects `createProjectV2Field` with *"Name cannot have a reserved value"*. Because
   > `setup-project.sh` runs under `set -euo pipefail` and `provision.sh` treats its failure as a
   > hard failure, that rejection aborted the script before `Priority` was reached and broke
   > provisioning outright for **every** repo, greenfield and `--upgrade` alike. Found by running
   > against a live throwaway board, not by review — nothing in `gh project field-create --help`
   > documents the reserved-name list. Evidence:
   > [`verification/results/TICKET-067-live-board.md`](verification/results/TICKET-067-live-board.md).
   > If you add a field of your own, smoke-test it against a throwaway board first.
   >
   > **Fields are permanent.** The upgrade path is additive-only: it can create a missing field but
   > can never remove or retype one, and renaming a single-select option does not migrate existing
   > cards. Add a field only when something actually writes it. `Gate Status` and `Repo` were
   > considered and deliberately not created — see
   > [`templates/factory/.github/PROJECTS.md`](templates/factory/.github/PROJECTS.md).

   **Then finish the board by hand — these are manual UI steps.** The GitHub API and `gh` expose
   **no** way to create Projects v2 **views**, per-column **WIP limits**, or the built-in **workflow
   automations** (auto-add, auto-archive), and **no** way to create **discussion categories**. No
   script attempts them, and any doc claiming a script does is wrong. Once, after step 4:
   1. a **Board view grouped by `Stage`** — the primary floor view;
   2. a **Table view grouped by `Spec`** — traceability across a delivery;
   3. **WIP limits** per Stage column, if you want them;
   4. **built-in automations** (auto-add items, auto-archive on close);
   5. **discussion categories** on the host repo — the `DISCUSSIONS-*.md` charters in
      `templates/factory/.github/` document intent for a human to type in; they are not
      provisioning input.

   Background and the full script-provisioned-vs-UI-only split:
   [`docs/board-and-gates.md`](docs/board-and-gates.md).

## Landing workflow templates (`land-workflows.sh`)

The factory **never auto-ships workflows** (ADR-0008): `bootstrap-project.sh` and
`setup-repo.sh` both refuse to stage anything under `.github/workflows/`. Landing one is an
explicit operator act, like `enable-audit.sh`.

```bash
cd /path/to/<target-repo>-checkout
.ai/templates/factory/scripts/land-workflows.sh <owner/repo>            # dry run (default)
.ai/templates/factory/scripts/land-workflows.sh <owner/repo> --apply
```

Dry run is the default; it prints the plan and writes nothing. `--apply` copies the files
into the checkout for **you** to commit and open a PR — the verb never commits, pushes,
branches or merges.

> **Atomic units — a partial land is refused, not warned about.** `floor motor` and
> `stage-arbiter.yml` are one unit (SPEC-017). After the arbiter cutover the motor no longer
> drives on `issues: labeled`, so a repo holding the new motor **without** the arbiter has a
> belt that drives on nothing and is driven by nothing — and the documented recovery
> ("re-add the stage label") is exactly what stops working. Landing either alone exits
> non-zero. A repo already in that half-landed state is reported as **BROKEN BELT** on every
> run, with the remediation command, whatever you were actually asking for.

Re-running is a no-op when the files already match, so it is safe in a loop.

## Per repo

Run these **once per code repo** you onboard. `bootstrap-project.sh` seeds the local
convention files; the **enforcement layer (CI/deploy workflows + rulesets) is installed at
activation** by `setup-repo.sh` / [`ACTIVATE.md`](templates/factory/ACTIVATE.md) — see the
precise split below.

> **One-command path (`scripts/provision.sh`).** The per-repo steps below (bootstrap →
> repo-create → JOIN the org Project → `setup-repo.sh` gates → seed the board-sync
> `sync-issues.yml` workflow) are collapsed into a single idempotent orchestrator:
> `scripts/provision.sh "$ORG/<repo>"` (add `--dry-run` to preview with zero mutations,
> `--upgrade` to retrofit an existing repo). It is the engine-only entry point — top-level
> `scripts/`, never vendored — documented in
> [ADR-0012](docs/decisions/ADR-0012-factory-provisioning-automation.md) and
> [`ACTIVATE.md`](templates/factory/ACTIVATE.md) Phase 3. The manual steps below remain the
> by-hand breakdown if you need to run a single leg in isolation; `provision.sh` just runs
> them in the right order and adds the workflow seed (which needs a `workflow`-scope push
> credential — the `github-your-org:` SSH alias / App token, not the global HTTPS token).

1. **Bootstrap the method** — from the factory repo root:
   ```bash
   ./scripts/bootstrap-project.sh /path/to/<repo>-delivery
   ```
   *Artifact:* `.ai/` method, `.claude/` agents+commands, lifecycle dirs, ledger, knowledge
   YAMLs, a stub `Makefile`, **and the seeded conventions** — `.github/ISSUE_TEMPLATE/`,
   `.github/PULL_REQUEST_TEMPLATE.md`, and a root `CODEOWNERS`.
   > **Seeded-vs-activation boundary (important).** `bootstrap-project.sh` seeds only the
   > inert **conventions** (issue/PR templates + `CODEOWNERS`) locally. It deliberately does
   > **not** copy anything under `.github/workflows/` — the CI/deploy workflows and the
   > rulesets are installed later at activation by `setup-repo.sh` / `ACTIVATE.md`. This keeps
   > `PROJECTS_TOKEN`-bearing workflows and the off-by-default autonomy workflow out of a bare
   > repo until the deliberate, secret-gated activation step.
   >
   > **Remediation — repos bootstrapped between 2026-07-17 and 2026-07-18.** A short-lived
   > state of this factory copied the *authoring* templates (`templates/specs/`,
   > `templates/tickets/`) into `.github/ISSUE_TEMPLATE/`, plus a `.github/DISCUSSIONS/`
   > directory GitHub never reads and a `.github/PROJECTS.md` carrying a non-canonical Stage
   > vocabulary. Fixed by TICKET-066. Because these are seeded **copy-if-absent** (ADR-0005),
   > **re-running `bootstrap-project.sh` will NOT replace them** — a repo that received them
   > keeps them permanently until someone deletes them by hand. Check and clean up with:
   > ```bash
   > cd /path/to/<repo>-delivery
   > ls .github/ISSUE_TEMPLATE/            # expect only FEATURE-REQUEST.md, BUG.md
   > rm -f .github/ISSUE_TEMPLATE/spec-template.md .github/ISSUE_TEMPLATE/ticket-template.md
   > rm -rf .github/DISCUSSIONS .github/PROJECTS.md
   > ```
   > Symptom if left in place: authors picking "SPEC-XXX"/"TICKET-XXX" from the *issue* form
   > produce artifacts whose front-matter fails the `factory-naming` required check.
2. **Wire the Makefile** — fill in the stub's real commands. CI runs `make lint`, `make
   test`, `make build` as required checks (the stub seeds these plus `test-local` /
   `test-docker` for the delivery flow; all are graceful no-ops until you wire them).
3. **Apply gates in report-only mode** — `templates/factory/scripts/setup-repo.sh "$ORG/<repo>"`.
   Rulesets ship `enforcement: evaluate` so onboarding never blocks an in-flight PR.
   *Artifact:* labels, ruleset + naming ruleset (in `evaluate`), staging/production
   environments. **Greenfield (Path A):** also copy the shipped `ci.yml`/`deploy.yml`.
   **Existing repo (Path B):** keep the existing CI; `setup-repo.sh` reconciles the ruleset's
   required checks to it.
4. **Land the `.github/` + CODEOWNERS state** — bootstrap already seeded these locally
   (step 1); for an **existing repo**, open a PR to bring them in (`setup-repo.sh` prints the
   paths). Workflows are added here at activation, not by bootstrap.
   > **Optional auto-PR:** `setup-repo.sh "$ORG/<repo>" --stage-files` opens this governance
   > PR for you (clones the repo, copies the governance subset — `CODEOWNERS`, issue/PR
   > templates — **copy-if-absent**, never workflows, and **opens** a PR). It is **open-only**
   > (never merges/approves/force-pushes) and best-effort (ADR-0007). Opt-in, default OFF —
   > without the flag the manual copy above stays the behaviour.
5. **Open a test PR and verify** — confirm the required checks run (`lint`, `test`, `build`,
   `verification`, `audit` per the ruleset variant) and that the PR requests review from the
   expected CODEOWNERS team, while rulesets are still in `evaluate`.
6. **Set required reviewers on `production`** — Settings ▸ Environments ▸ production. This is
   a manual, one-time step; without it the deploy gate is non-enforcing.
7. **Flip rulesets to enforcing** — only after the test PR is green **and** `setup-teams.sh`
   has resolved the CODEOWNERS teams:
   ```bash
   ENFORCEMENT=active templates/factory/scripts/setup-repo.sh "$ORG/<repo>"
   ```
   The re-run is idempotent — it **updates the existing ruleset in place** (the earlier
   `evaluate` run created it; this flips `enforcement`). Or flip in Settings ▸ Rules ▸
   Rulesets. Confirm on GitHub that `main` now requires the checks + 1 CODEOWNERS approval
   and blocks force-push.

> **Optional — require the independent audit verdict (`independent-audit`).** *Only if* the
> independent DeepSeek audit gate has been landed on this repo (via
> `scripts/enable-audit.sh`) **and** `deepseek audit key` is set, you MAY add the
> `independent-audit` required-status-check context so a DeepSeek **Fail** verdict *blocks
> merge*. This is the single authoritative place for that flip — do not invent a parallel
> procedure.
> - **Require it only where the workflow is landed (coupling).** Add `independent-audit`
>   **only** to a repo that has landed `independent-audit.yml`; requiring a context that
>   never reports dead-blocks every PR. Fork / no-secret PRs still report **green** (the
>   secret-presence skip is a green *step*, not a job-level skip), so enabling it does not
>   wedge external contributors.
> - **How (emit-and-review, not a new procedure).** Add the context by reconciling the
>   ruleset — the same emit-and-review flow already used for the other CI checks:
>   `reconcile-ci-checks.sh` emits the JSON, you review it, then apply via
>   `RULESET=<path> setup-repo.sh "$ORG/<repo>"` at `evaluate`. `enable-audit.sh` prints the
>   exact invocation. The factory **prints/prepares**; you review and apply.
> - **The `evaluate → active` flip is yours (step 7).** The context lands at
>   `enforcement: evaluate` (report-only) and flips to `active` **with the rest of the
>   ruleset** at step 7 — an operator-performed, operator-confirmed action. The factory
>   never runs `ENFORCEMENT=active` and never flips its own gate.
> - **Validate before the flip (Phenomenon B — resolved for the PR path).** Phenomenon B —
>   where the audit's report-commit moved the PR head to an unchecked SHA and left the required
>   check pending forever — is now **resolved for the PR path** by
>   [ADR-0010](docs/decisions/ADR-0010-audit-no-branch-commit.md) / SPEC-012: the `--pr` driver
>   no longer commits its report to the PR branch (it records the verdict via the sticky PR
>   comment + the Actions run summary), so the PR head no longer moves and the required check is
>   not left pending on an unchecked head. The pre-flip validation is therefore **expected to
>   pass**. Still confirm it on the live test PR (step 5) as a sanity check BEFORE flipping to
>   `active`: verify the `independent-audit` check reports on the PR's own head commit. (This is
>   a confirmation now, not an expected blocker. The manifestation and its fix are recorded in
>   [ADR-0009](docs/decisions/ADR-0009-required-audit-check-collision.md) / TICKET-040 →
>   [ADR-0010](docs/decisions/ADR-0010-audit-no-branch-commit.md) / SPEC-012. The issue/motor
>   driver still commits its report — a different flow, not the required-check path.)
> - **It raises the floor — it is not the whole floor (honest framing).** A required
>   `independent-audit` check RAISES the merge floor but is **not** the sole control: the
>   verdict is derived from model output over an attacker-influenceable diff, so a
>   prompt-injected diff could steer the model to emit a **Pass**. Human CODEOWNERS approval
>   and the other required checks (`lint`/`test`/`build`/`verification`) remain the enforcing
>   gate. The independent verdict adds a machine gate; it does not replace human review.

The full per-repo go-live sequence, with every check spelled out, lives in
[`ACTIVATE.md`](templates/factory/ACTIVATE.md); this runbook is the ordering, that is the detail.

## Upgrading an existing instance

A repo that was provisioned before a factory change — new board fields, new labels, a reconciled
ruleset — is brought current with **one command**:

```bash
./scripts/provision.sh "$ORG/<repo>" --upgrade                 # GitHub-side legs only
./scripts/provision.sh "$ORG/<repo>" --upgrade --checkout /path/to/<repo>-delivery
./scripts/provision.sh "$ORG/<repo>" --upgrade --dry-run       # print the plan, mutate nothing
```

**There is no separate upgrade script, and none is wanted.** `provision.sh --upgrade` *is* the
upgrade path. It already implements every leg (`scripts/provision.sh:261-317`):

| Leg | What runs | Notes |
|---|---|---|
| 1. Existence check | `gh repo view "$REPO"` | **Hard fail** if the target does not exist — `--upgrade` never creates a repo. |
| 2. Local `.ai/` method bundle | `scripts/upgrade-project.sh "$CHECKOUT_DIR"` | **Runs only when `--checkout` is supplied.** Without it there is no local tree to refresh and the leg is **skipped entirely** (loudly, on stdout); the GitHub-side legs still run. |
| 3. Board | `templates/factory/scripts/setup-project.sh` | JOINs the org Project, never recreates it; adds any field missing from step 4 above. |
| 4. Gates | `templates/factory/scripts/setup-repo.sh "$REPO"` | Labels, rulesets, CODEOWNERS, environments, naming ruleset. |
| 5. Board-sync seed | `sync-issues.yml` | **Best-effort**: a failure warns loudly and does **not** fail the run. |

Every leg is idempotent — re-running an already-current repo reports "already present" and mutates
nothing. That is the whole design: upgrade is not a distinct operation, it is provisioning run
again with repo creation replaced by an existence check.

> **The tempting mistake — read this before you go looking for the missing piece.**
> `scripts/upgrade-project.sh` read on its own looks like the whole upgrade path, because it is
> self-contained and clearly named. **It is not.** It is *only* leg 2, the local-filesystem method
> bundle, invoked *by* the orchestrator — it makes no GitHub API calls, touches no board and no
> ruleset. Reading it in isolation makes the GitHub leg look absent and invites writing a second
> script to supply it. Do not: that script would duplicate legs already shipped in `provision.sh`.
> [ADR-0005](docs/decisions/ADR-0005-install-file-ownership-boundary.md) rejected a standalone
> `upgrade-existing` verb as needless surface, and
> [ADR-0012 §(d)](docs/decisions/ADR-0012-factory-provisioning-automation.md) rejected
> per-sub-script `--dry-run`/`--apply` interfaces for the same reason. Both decisions stand.

**Dry-run ergonomics — the convention differs by script class, deliberately.** Know which you are
holding before you run it:

| Class | Default | Opt out with | Example |
|---|---|---|---|
| Orchestrators | **apply** | `--dry-run` | `scripts/provision.sh` (`:321-327`) |
| One-shot bootstrap verbs | **dry-run** | `--apply` | `setup-host-repo.sh` (`:31-33`) |
| Leaf scripts | apply — **neither flag** | — | `setup-project.sh`, `setup-repo.sh` |

Leaf scripts mutate on invocation and have no preview mode; `--dry-run` is gated at the
orchestrator, which then invokes **no** sub-script at all (ADR-0012 §(d)).

**What upgrade cannot fix.** Two classes of state are out of reach:

- **Manual UI state** — views, WIP limits, Project automations, discussion categories. Re-check
  these by hand after an upgrade that adds a Stage option or field; see step 4.
- **Convention files already present in the target repo.** The factory seeds `.github/` templates
  and `CODEOWNERS` **copy-if-absent** ([ADR-0005](docs/decisions/ADR-0005-install-file-ownership-boundary.md) §2),
  so it can never update or remove one that already exists — by design, so the factory cannot
  clobber a repo's own conventions. A repo bootstrapped from the broken tree keeps the bad files
  until a human deletes them; **`--upgrade` will not remove them.** The exact remediation commands
  are in the "Per repo" step 1 note above.

## "Am I done?" checklist

You are **not** done until every box is checked — a repo can look onboarded while its gates
are still non-enforcing, which is a net regression:

- [ ] `./scripts/check-readiness.sh` exits 0.
- [ ] Builder App installed; a `…[bot]` PR was opened and a **human** approved it.
- [ ] `setup-teams.sh` passed — every CODEOWNERS team exists and is non-empty.
- [ ] `bootstrap-project.sh` seeded `.github/` templates + root `CODEOWNERS` in the repo.
- [ ] A test PR ran the required checks green and requested the expected CODEOWNERS review.
- [ ] **Rulesets are `active`** (not `evaluate`): `main` requires the checks + 1 CODEOWNERS
      approval and blocks force-push.
- [ ] **The `production` environment has a required reviewer set.**
- [ ] `/feature-delivery` runs and stops at the human gate.

If either of the last two enforcement boxes is unchecked, the repo is **not** gated — go back
to the per-repo steps and finish the `evaluate → active` flip and the production reviewer.
